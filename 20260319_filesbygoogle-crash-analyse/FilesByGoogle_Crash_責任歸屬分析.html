Files by Google Crash 責任歸屬詳細分析

Case #2 — 多層級共同責任分析與行動計劃

日期：2026-03-19
裝置：Thorpe DVT3 (T70)
問題：USB 大量複製時 OOM Kill

結論：多層級共同責任，主要責任在 Linux Kernel 與 Android Framework

Files by Google 的閃退並非單一元件的 bug，而是從 Kernel 記憶體管理、Android LMKD、FUSE 架構、App 設計到裝置配置的多層級系統性問題。以下是各層級的責任佔比與詳細分析。

責任分配總覽

Kernel 40%
LMKD 25%
FUSE 20%
App 10%
ODM 5%
責任方佔比具體問題應該做什麼
Linux Kernel 40% OOM Killer 不應殺前景 App；dirty page 管理不當 調整 dirty_ratio、優化 OOM Killer 的 adj 判斷
Android Framework (LMKD) 25% LMKD 未保護前景 I/O App 向 Google 回報，改善 LMKD 對前景 I/O 的保護
Android FUSE 架構 20% FUSE 雙重 buffer 大幅增加記憶體消耗 Google 長期架構問題，短期無法改變
Files by Google 10% 缺乏 memory-aware 複製策略 建議 Google 實作 chunked I/O
ODM/BSP (Thorpe) 5% RAM/ZRAM 配置、dirty_ratio 參數 調整 kernel 參數、評估增加 ZRAM
1. Linux Kernel (記憶體管理) — 主要責任
40%

為什麼是 Kernel 的問題

  • OOM Killer 選擇了正在前景運行的 App 來殺 — 這是不合理的行為
  • 正常情況下，OOM Killer 應優先殺 background/cached 進程，而非 foreground TOP 進程
  • 同一秒內回收 64~71 個進程，說明記憶體管理已完全失控，不是漸進式 graceful degradation

具體問題

問題說明
target_dirty_ratio=80允許 80% 可回收記憶體被 dirty pages 佔用，太高了
USB I/O dirty pages 擠壓大量 USB I/O 導致 dirty pages 佔用過多 page cache
kswapd 搶救不及持續 14% CPU，kernel 一直在回收但來不及
OOM adj 判斷失誤前景 TOP 進程的 oom_adj 保護機制失效

Log 證據

oom_reaper: reaped process 4796 (.apps.nbu.files) ← 前景 App 被殺 reason=3 (LOW_MEMORY) subreason=0 (UNKNOWN) kswapd0:1: 14% CPU, kswapd-1:0: 13% CPU ← 持續高壓 256+ oom_reaper events ← 大量進程被回收

建議修復

  • 調低 target_dirty_ratio 從 80 → 40~50
  • 調低 dirty_reclaim_rate 從 0.5 → 0.3
  • 優化 OOM Killer 的 oom_adj 評分，確保前景 TOP 進程有更強的保護
2. Android Framework (LMKD / ActivityManager) — 次要責任
25%

為什麼是 Framework 的問題

  • LMKD 的職責是在記憶體不足時有序地殺進程
  • 應保護 oom_adj 為前景 (fg TOP) 的進程不被殺
  • 保護機制失效 — 前景正在複製檔案的 App 還是被殺了

具體問題

問題說明
LMKD reinitlmkd --reinit 在 crash 前執行，LMKD 自己也出了狀況
前景 I/O 無保護Framework 無「前景 App 正在進行大量 I/O」的額外保護
mem-pressure-event 失效頻繁觸發但未能有效阻止前景 App 被殺

Log 證據

init: processing action (lmkd.reinit=1) ← LMKD 被重新初始化 init: starting service 'exec 37 (/system/bin/lmkd --reinit)' Rescheduling restart of crashed service ... for mem-pressure-event ← 記憶體壓力事件頻繁

建議修復

  • 向 Google Issue Tracker 回報 Bug
  • 分類：Android > Framework > ActivityManager 或 Storage
  • 強調 LMKD 應保護正在進行 I/O 的前景 App
3. Android Storage / FUSE — 次要責任
20%

為什麼是 FUSE 的問題

  • Android 11+ 強制使用 FUSE 掛載 /storage/emulated
  • FUSE 架構比 direct I/O 多消耗約 2-3 倍記憶體
  • 每次 read/write 需要 kernel → userspace (MediaProvider) → kernel 的雙重 buffer
  • 複製 24.7GB 通過 FUSE 的記憶體開銷遠大於直接寫入

FUSE I/O 路徑

USB Storage
→
Kernel VFS
→
FUSE Daemon
(MediaProvider)
額外 buffer + context switch
→
Kernel VFS
→
Internal Storage

Google 的架構決策：為 Scoped Storage 安全性犧牲 I/O 效率

影響說明

  • 這是 Google 為了 scoped storage 安全性的架構決策
  • 短期內無法改變，但 Google 每個 Android 版本都在優化 FUSE 效能
4. Files by Google (App) — 輕微責任
10%

為什麼 App 也有一點責任

  • 作為執行大量檔案複製的 App，理應實作 memory-aware 的複製策略
  • 可以使用 ComponentCallbacks2.onTrimMemory() 響應記憶體壓力
  • 可以實作 chunked I/O，分批複製而非一次性大量寫入
  • 可以在記憶體壓力時主動暫停複製、提示使用者

但為什麼責任很輕

App 被 OOM Killer 直接殺掉

連 onTrimMemory() 都來不及收到就死了，App 層能做的有限。這是系統層級的暴力終止，App 無法防禦。

5. ODM / BSP 團隊 (Thorpe 裝置) — 配置責任
5%

配置問題

配置項現況問題
RAM7.1 GB對 USB 大量檔案傳輸的使用場景可能偏低
ZRAM4 GB在大量 I/O 時不足以緩衝
target_dirty_ratio80太高，應降至 40~50
dirty_reclaim_rate0.5可以更積極回收，建議 0.3

行動計劃

Phase 1 立即行動（ODM 可自行處理）

  • 調低 target_dirty_ratio 從 80 → 40~50
  • 調低 dirty_reclaim_rate 從 0.5 → 0.3
  • 評估增加 ZRAM 從 4GB → 6GB
  • 在 Thorpe 裝置上測試以上調整
  • 驗證：反覆進行 USB → 內部儲存複製 24.7GB 測試，確認不再閃退

Phase 2 需要 Google 介入

  • 向 Google Issue Tracker 提交 Bug Report
  • 附上兩份完整 bugreport 作為證據
  • 強調 LMKD 應保護正在進行 I/O 的前景 App
  • 建議 Google Files 團隊實作 chunked I/O

Phase 3 長期追蹤

  • 追蹤 Google 對 FUSE I/O 效率的改善
  • 確認是否在後續 Android 版本中修復 LMKD 保護機制
  • 監控 Thorpe 裝置上調整參數後的長期穩定性

比對線上已有的 Android OOM/LMK 技術文章，驗證本分析的正確性並標示差異。

1. AndroidPerformance — 低記憶體導致卡頓
最高相關

androidperformance.com — Android Jank Due To Low Memory

比較項目該文章本 Case

I/O 阻塞Uninterruptible Sleep 從 130ms 飆到 750msUSB→FUSE 大量 I/O 加劇問題

該文章適用於本 Case 的建議

  • 使用 cgroup blkio 限制背景 I/O
  • 改進 LMK 避免死亡-重啟循環
  • 調整 read_ahead_kb 避免過度預讀
2. Android 官方 — Memory Management
高度相關

developer.android.com — Memory Management

  • kswapd 在記憶體不足時將 dirty pages 移入 ZRAM 壓縮
  • 系統先透過 onTrimMemory() 通知 App，再啟動 LMK
  • 「Killing the foreground app looks like an application crash」 — 與本 Case 完全吻合
  • 官方確認前景 App 應是最後被殺的，但本 Case 中保護機制失效
3. Google AOSP — LMKD 官方文件
高度相關

source.android.com — Low Memory Killer Daemon

  • LMKD 使用 kernel 的 vmpressure 事件或 PSI 監控記憶體壓力
  • 使用 memory cgroup 限制各進程的記憶體使用
  • 根據 oom_score_adj 決定殺哪個進程
  • 與本 Case 關係：lmkd --reinit 在 crash 前觸發，說明 LMKD 自身出問題
4. learn-house — LowMemoryKiller 防止應用被殺

等級adj 值說明
