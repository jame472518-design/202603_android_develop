📄 問題描述

回報者測試時使用 SanDisk USB-C 隨身碟（32GB/64GB），將兩個資料夾（各含五個 2.47GB ZIP 檔）從 USB 複製到內部儲存空間 Documents。重複填滿內部記憶體（約 85GB 可用，可複製 3 次約 25GB 的資料）然後全選刪除、重新開機，反覆操作。在約 10~15 次填滿操作過程中，Files by Google 發生了 2 次閃退/消失。

回報者懷疑這是 Android 層級的問題，因為在 Akita 專案上進行相同測試也出現類似現象。

⚠ 關鍵結論：Files by Google 並非 App Crash，而是被 Linux Kernel OOM Killer 強制終止

兩次事件的 kill reason 均為 reason=3 (LOW_MEMORY)，由 kernel oom_reaper 直接回收進程記憶體。這不是 Files by Google 本身的 bug，而是系統級記憶體不足導致前景 App 被殺。

🔍 Root Cause 分析

  • 直接原因：Linux Kernel OOM Killer 強制終止 Files by Google 進程
    兩次事件均記錄 reason=3 (LOW_MEMORY)，由 oom_reaper 直接回收。這不是 App crash (exception/ANR)，而是系統級的記憶體不足回收。
  • 記憶體壓力來源：大檔案複製產生的 Page Cache 壓力
    從 USB 複製 10 個 2.47GB 檔案 (~24.7GB) 到內部儲存時，Linux kernel 需要大量 page cache 來緩衝 I/O。kswapd 持續以 14% CPU 運行，表示 kernel 一直在回收記憶體。usb-storage 也消耗 3.1% CPU。
  • FUSE 檔案系統加劇記憶體消耗
    Android 使用 FUSE (Filesystem in Userspace) 掛載 /storage/emulated，每次 I/O 需要額外的 kernel-to-userspace context switch 和 buffer copy。ExternalStorageServiceImpl (MediaProvider) 也在運行中。大量檔案複製通過 FUSE 會顯著增加記憶體使用。
  • 系統 RAM 不足以應付此工作負載
    裝置僅有 ~7.1 GB RAM + 4 GB ZRAM swap。複製 24.7GB 大檔案時，page cache 和 dirty pages 的需求遠超可用記憶體。在 crash 前的一秒內，有 64~71 個進程被同時回收，顯示系統已進入極度記憶體不足狀態。
  • 反覆填滿/刪除加劇碎片化
    測試步驟要求反覆填滿儲存空間再刪除，這會導致 f2fs/ext4 檔案系統碎片化，增加 metadata 記憶體使用量，並使 I/O 效率降低。

📝 結論與建議

這是 Android 問題還是 Files by Google 問題？

報告者的懷疑是正確的 — 這主要是 Android/Kernel 層級的記憶體管理問題，而非 Files by Google 的 App bug。Files by Google 是被系統 OOM Killer 殺掉的受害者。

建議處理方案

方案層級說明負責方
1. 向 Google 回報 Bug Android / Google 回報至 Google Issue Tracker，分類為 Android > Framework > Activity Manager 或 Storage。 附上兩份 bugreport，說明大量 USB-to-internal 檔案複製觸發 OOM kill foreground app。 Google 可能需要優化 FUSE I/O 路徑的記憶體使用或 LMK 對前景複製操作的保護。 ODM -> Google
2. 調整 LMK 參數 Kernel / Vendor 調整 lmkd 參數，對正在進行大量 I/O 的前景 App 提供更高的保護優先級。 考慮調低 dirty_reclaim_rate 和 target_dirty_ratio（目前為 80%）。 BSP / Kernel 團隊
3. 限制 Page Cache 使用量 Kernel 設定 vm.dirty_ratio 和 vm.dirty_background_ratio 為較低值， 避免大量 I/O 時 dirty pages 佔用過多記憶體。 BSP / Kernel 團隊
4. 考慮增加 ZRAM 大小 Vendor 目前 ZRAM 為 4GB，可評估增加至 6GB 以提供更多 swap 緩衝。 BSP 團隊
5. Files by Google 應用層優化 App (Google) 建議 Google 在大量複製操作時實作 chunked I/O 和 memory-aware throttling， 減少單次複製操作的記憶體佔用。可使用 ComponentCallbacks2.onTrimMemory() 響應。 Google (Files 團隊)
